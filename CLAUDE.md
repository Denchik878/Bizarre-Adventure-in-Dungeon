# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A 2D pixel-art Metroidvania/dungeon platformer built in **Unity 6 (6000.0.25f1)** using the **Universal Render Pipeline (2D)**. Gameplay is built on top of the third-party *Metroidvania Controller* asset; the *Cainos – Pixel Art Platformer Dungeon* asset supplies the dungeon tileset and is art-only (no scripts).

## Building, running, and testing

There is no command-line build/test workflow checked in — this is a standard Unity Editor project.

- **Open / run:** Open the project in Unity 6000.0.25f1 (versions must match; see `ProjectSettings/ProjectVersion.txt`). Press Play in the Editor.
- **Main playable scene:** `Assets/MetroidvaniaController/Scenes/Demo.unity` (the full gameplay sample). `Assets/Cainos/Pixel Art Platformer - Dungeon/Scene/SC Demo Scene.unity` is the tileset showcase being integrated.
- **Tests:** None exist. There are no `.asmdef` files and no test assemblies, so all gameplay code compiles into the default `Assembly-CSharp`. The `Unity.*.csproj` / `Assembly-CSharp.csproj` files at the repo root are auto-generated by Unity — do not hand-edit them.
- **Player controls (hard-coded, legacy Input Manager):** Arrow keys / `Horizontal` axis to move, `Z` jump (hold to double-jump), `C` dash, `X` melee attack, `V` throw weapon.

## Code architecture

All gameplay scripts live under `Assets/MetroidvaniaController/Scripts/`, grouped as `Player/`, `Enemies/`, and `Environment/`. There is no namespacing or DI — components find each other via `GetComponent`, `transform.Find`, and `GameObject.Find`.

**Player (input → physics split):**
- `PlayerMovement` reads input every `Update`, buffers `jump`/`dash` flags, and forwards them to `CharacterController2D.Move()` in `FixedUpdate`.
- `CharacterController2D` is the core physics/state machine: grounded/wall checks via `Physics2D.OverlapCircleAll` against `m_GroundCheck`/`m_WallCheck` transforms, plus double-jump, dash, wall-slide, wall-jump, knockback, i-frames, and death. It drives the `Animator` via bool params (`IsJumping`, `IsWallSliding`, `IsDashing`, `Hit`, `IsDead`, …) and exposes `UnityEvent`s `OnFallEvent`/`OnLandEvent`.
- `Attack` is a separate component (melee + throwable instantiation). Melee damage is applied from an **animation event** calling `DoDashDamage()`, not from code timing.

**Damage convention:** Damage is delivered two ways — keep both in mind when changing combat:
- Enemies/allies receive damage via `gameObject.SendMessage("ApplyDamage", dmgValue)` where the sign of `dmgValue` encodes knockback direction.
- The player receives damage via a direct `CharacterController2D.ApplyDamage(amount, attackerPosition)` call (position drives knockback direction).

**Enemies:**
- `Enemy` — simple patroller; uses `FallCheck`/`WallCheck` child transforms to turn at ledges/walls, damages the player on `OnCollisionStay2D`.
- `Ally` — companion AI with a weighted random decision loop (run/jump/dash/range/idle) that targets the GameObject literally named `"DrawCharacter"` (resolved via `GameObject.Find`). If you rename that object, allies stop working.

**Camera / feel:** `CameraFollow` follows the player and exposes `ShakeCamera()`, which `Attack` calls on a successful hit.

**Cross-cutting conventions:**
- Cooldowns, i-frames, stuns, and death sequences are implemented as `IEnumerator` coroutines with `WaitForSeconds`, not timers in `Update`.
- Rigidbody2D uses the Unity 6 API `linearVelocity` (not the deprecated `velocity`) — match this in new code.
- Facing is flipped by negating `transform.localScale.x`; many direction checks multiply by `localScale.x`, so don't switch to sprite-flipping without auditing those.
- On player death, `CharacterController2D` reloads the active scene via `SceneManager.LoadSceneAsync`.

## Gotchas

- The `com.unity.inputsystem` package is installed but gameplay still uses the **legacy** `Input.GetKeyDown` / `Input.GetAxisRaw`. Don't assume the new Input System is wired up.
- `.cs` files have committed `.meta` files (GUIDs); when adding or moving scripts let Unity generate the `.meta`, and commit it alongside the script.
- Generated/derived folders (`Library/`, `Temp/`, `Logs/`, `obj/`) and the root `*.csproj`/`.sln` are not source of truth — `Assets/`, `Packages/manifest.json`, and `ProjectSettings/` are.
